# Резервное копирование и масштабирование

## Резервное копирование

Всё хранится в **одной базе данных PostgreSQL** — данные приложения, очередь
задач (River) и HTTP-сессии. Резервное копирование этой базы данных сохраняет
всё состояние системы.

```bash
# Dump
pg_dump "$DATABASE_URL" -Fc -f emergency_callback_$(date +%F).dump

# Restore (into an empty database)
pg_restore -d "$DATABASE_URL" --clean --if-exists emergency_callback.dump
```

Рекомендуется:

- Запланировать ежедневный `pg_dump` (cron/таймер systemd) и хранить копии вне
  хоста.
- Также держите под контролем версий вашу **конфигурацию Asterisk**
  (пользователь AMI, три пользовательских контекста и изменения trunk/маршрутов)
  и каталог `audios/` из репозитория — они необходимы для восстановления стороны
  телефонии.

Чего **нет** в базе данных: бинарный файл, `.env` (секреты), конфигурация
Asterisk и установленные аудиофайлы. Их резервируйте отдельно.

## Масштабирование параллелизма

Количество одновременных вызовов определяется **тремя** ограничениями —
побеждает наименьшее из них:

| Ограничение | Где | Примечания |
|-------|-------|-------|
| `RIVER_MAX_WORKERS` | `.env` | Максимальное число задач, которые worker выполняет одновременно. Каждый активный вызов = одна задача. |
| Размер пула БД | `DB_POOL_MAX_CONNS` | Должен с запасом превышать число воркеров; оставаться ниже `max_connections` PostgreSQL. |
| Каналы trunk | Asterisk / провайдер | Лимит одновременных каналов вашего SIP trunk. |

Чтобы повысить пропускную способность:

1. Увеличьте `RIVER_MAX_WORKERS`.
2. Убедитесь, что `DB_POOL_MAX_CONNS` и `max_connections` PostgreSQL достаточно
   высоки.
3. Подтвердите, что trunk допускает столько одновременных исходящих каналов.
4. Каждый вызов открывает **новое соединение AMI**; убедитесь, что AMI Asterisk
   (`authlimit`) и лимиты файловых дескрипторов справляются с пиковой нагрузкой.

Перезапустите worker после изменения `.env`:

```bash
sudo systemctl restart emergency-callback-worker
```

## Запуск нескольких воркеров

Вы можете запустить более одного процесса `worker` (например, на разных хостах)
против одной и той же базы данных — River координирует блокировку задач в
PostgreSQL, поэтому задачи не обрабатываются дважды. Держите число **веб**-
процессов умеренным (он лишь обслуживает UI и ставит задачи в очередь).
Суммарный параллелизм воркеров всё равно должен учитывать лимит каналов trunk.

## Ротация секретов

| Секрет | Эффект ротации |
|--------|--------------------|
| `SESSION_SECRET` | Все пользователи выходят из системы (сессии аннулируются). |
| `CSRF_KEY` | Формы, находящиеся в процессе отправки, отклоняются один раз (пользователи перезагружают страницу). Должен оставаться ровно 32 байта. |
| `AMI_SECRET` | Должен быть изменён **и** в `.env`, **и** у пользователя AMI Asterisk, затем перезапустите worker и перезагрузите Asterisk. |
| `ESKIZ_PASSWORD` | Обновите `.env`, перезапустите worker (клиент повторно аутентифицируется). |
| Пароль БД | Обновите роль в PostgreSQL и `DATABASE_URL`, перезапустите оба сервиса. |

## Выпуск новой сборки

```bash
# 1. Build
go build -o emergency-callback ./cmd/emergency-callback

# 2. Apply migrations (safe to run repeatedly)
./emergency-callback migrate up
river migrate-up --database-url "$DATABASE_URL"   # only if River added migrations

# 3. Swap the binary and restart
sudo systemctl restart emergency-callback-web emergency-callback-worker
```

Активные вызовы завершаются на старом worker до его выхода, если вы
останавливаете его корректно; для развёртываний без потерь сначала выполните
дренаж, приостановив создание новых обратных вызовов.

## Рекомендации по мониторингу

- **Проверка живости:** `curl -sf http://127.0.0.1:8000/users/login/`.
- **Прогресс worker:** наблюдайте за `journalctl -u emergency-callback-worker`.
- **Зависшие вызовы:** периодический `CleanupStaleCalls` финализирует всё, что
  застряло более чем на 30 мин; растущее число `failed`/`no_rating` указывает на
  проблему с trunk.
- **Глубина очереди:** проверяйте таблицы River в PostgreSQL (например, считайте
  строки в `river_job` по состоянию), чтобы увидеть отставание.
